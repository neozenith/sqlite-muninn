# Plan: WASM Integration + Dynamic DB Selector

> Merge the standalone `wasm/` demo into `viz/` and add a database picker so both
> the server-side (FastAPI) and client-side (WASM) explorers can target any
> database the demo_builder produces.

## Status Quo

| Component | Stack | DB loading |
|-----------|-------|------------|
| `wasm/` | Vanilla HTML/JS, CDN libs (Deck.GL, Cytoscape, Transformers.js) | Hardcoded `fetch("assets/3300.db")` → Emscripten FS |
| `viz/` | FastAPI + React (Vite/TS/Tailwind v4/shadcn), 3 explorer modes | Singleton connection via `MUNINN_DB_PATH` env var (default `benchmarks/kg/3300.db`) |
| `demo_builder` | Python package under `benchmarks/demo_builder/` | Outputs `{book_id}_{model}.db` files to a configurable `--output-folder` (currently `wasm/assets/`) |

### Key Observations

- The WASM demo runs SQLite **entirely in the browser** (Emscripten WASM build of muninn) — no server needed.
- The viz tool runs SQLite **server-side** via FastAPI — browser never touches SQLite directly.
- Both share similar visualizations (Deck.GL point clouds, Cytoscape graphs) but with completely different implementations (vanilla JS vs React components).
- `wasm/assets/` already contains multiple DBs (`3300_MiniLM.db`, `3300_NomicEmbed.db`, `39653_MiniLM.db`, etc.) but the WASM demo ignores all but `3300.db`.

---

## Goals

1. **Consolidate** the WASM demo into the viz/ app as a new "WASM Explorer" mode (fourth tab alongside VSS/Graph/KG Pipeline).
2. **Dynamic DB selection** — both the server-side viz modes and the WASM mode let users pick from all available demo databases.
3. **Single asset folder** — demo DBs live in one canonical location that both modes read from.
4. **Retire `wasm/` as a standalone app** — all demo UX lives under viz/.

## Architecture

### Database Discovery

A lightweight **manifest endpoint** and a **static manifest file** provide database listings to both server-side and client-side consumers.

```
demo_builder build --output-folder viz/public/demos/
                         │
                         ▼
            viz/public/demos/
            ├── manifest.json          ← generated by demo_builder
            ├── 3300_MiniLM.db
            ├── 3300_NomicEmbed.db
            ├── 39653_MiniLM.db
            └── ...
```

**`manifest.json`** schema (generated by demo_builder at build time):

```json
{
  "databases": [
    {
      "id": "3300_MiniLM",
      "book_id": 3300,
      "model": "MiniLM",
      "dim": 384,
      "file": "3300_MiniLM.db",
      "size_bytes": 32059392,
      "label": "Wealth of Nations + MiniLM (384d)"
    }
  ]
}
```

### Two Consumption Paths

| Path | Who reads manifest | How DB is loaded |
|------|-------------------|------------------|
| **Server-side** (VSS/Graph/KG modes) | FastAPI reads `manifest.json` from disk, exposes `GET /api/databases` | User selects DB → `POST /api/databases/select` → server reopens SQLite connection |
| **Client-side** (WASM mode) | React fetches `manifest.json` directly (static file in `public/`) | User selects DB → browser `fetch()`es the `.db` file → writes to Emscripten FS |

## Implementation Phases

### Phase 1: Database Manifest Generation

**Goal:** demo_builder outputs a `manifest.json` alongside the `.db` files.

**Changes:**
- `benchmarks/demo_builder/manifest.py` — add `write_manifest_json(output_folder)` function that scans for completed `.db` files and writes `manifest.json` with metadata (book title, model, dim, size, chunk count)
- `benchmarks/demo_builder/cli.py` — call `write_manifest_json()` after every successful build
- New canonical output folder: `viz/public/demos/` (Vite serves `public/` as static assets)

### Phase 2: Server-Side DB Switching

**Goal:** The viz FastAPI backend can switch databases at runtime.

**Changes:**
- `viz/server/config.py` — add `DEMOS_DIR` pointing to `viz/public/demos/`
- `viz/server/routes/databases.py` — new router:
  - `GET /api/databases` — returns manifest (list of available DBs with metadata)
  - `POST /api/databases/select` — accepts `{ "id": "3300_MiniLM" }`, closes current connection, reopens with new DB path
- `viz/server/services/db.py` — refactor `get_connection()` to support runtime reconnection (close old conn, open new one, re-discover schema)
- Frontend: `DatabaseSelector` component (shadcn Select/Combobox) in the header, shared across all modes

### Phase 3: WASM Explorer Mode in viz/

**Goal:** Port the WASM demo's functionality into a React component within viz/.

**Changes:**
- `viz/frontend/src/components/WasmExplorer.tsx` — new mode component that:
  - Loads the muninn WASM module (Emscripten JS/WASM files from `public/wasm/`)
  - Fetches the selected `.db` from `public/demos/` and writes to Emscripten FS
  - Three-panel search UI (FTS, HNSW/Deck.GL, Graph/Cytoscape) — ported from `wasm/script.js` into React components
  - Transformers.js for in-browser embedding (same CDN approach, or bundled)
- `viz/public/wasm/` — Emscripten build artifacts (`a.out.js`, `a.out.wasm`) moved here
- New nav tab: "WASM Explorer" alongside existing modes
- The `DatabaseSelector` in the header also drives which DB the WASM mode loads

### Phase 4: Cleanup & Migration

**Goal:** Retire standalone `wasm/` demo, update build pipeline.

**Changes:**
- Move `wasm/assets/*.db` → `viz/public/demos/` (or regenerate via demo_builder)
- Move `wasm/build/` artifacts → `viz/public/wasm/`
- Update `wasm/Makefile` targets to build into `viz/public/wasm/`
- Update root `Makefile` if needed
- Remove or archive `wasm/index.html`, `wasm/script.js`, `wasm/styles.css` (the React port replaces them)
- Update `wasm/README.md` with redirect to viz/

## Key Decisions to Make

1. **DB file serving for WASM mode** — Vite's `public/` folder serves static files directly. For large DBs (30-50MB), this is fine for local dev. For deployment, may want a CDN or lazy-loading strategy. Punt for now?

2. **Transformers.js bundling** — The WASM demo uses CDN `<script type="module">` imports. In the React app, should we `npm install @huggingface/transformers` and bundle it, or keep the CDN approach via a dynamic import? Bundling is cleaner but increases bundle size.

3. **WASM module loading in React** — Emscripten modules expect global `Module` setup. Need a React-friendly wrapper (useEffect + ref pattern). The `wasm/script.js` is ~900 lines of vanilla JS — incremental porting vs full rewrite?

4. **Shared vs separate DB connections** — When user switches DB in the header, should it switch for ALL modes simultaneously, or per-mode? Simpler to make it global (one DB at a time).

---

## Out of Scope

- Deploying to a public URL (local dev only for now)
- Streaming/chunked DB loading for very large databases
- Multi-database comparison views
- Authentication or access control on the DB selector
