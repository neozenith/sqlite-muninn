# Colab-Ready Example Notebooks

**Status:** Draft spec for review
**Date:** 2026-03-17

## Goal

Make every example in `examples/` launchable as a Google Colab notebook via a badge in its `README.md`, while preserving the ability to run each example as:

1. A standalone Python script (`python examples/foo/example.py`)
2. A local Jupyter notebook
3. A one-click Google Colab session

## Requirements

### R1: Colab Badge in Every README (Critical)

Each `examples/**/README.md` MUST contain an "Open in Colab" badge that links to the correct `.ipynb` for that example:

```markdown
[![Open In Colab](https://colab.research.google.com/assets/colab-badge.svg)](https://colab.research.google.com/github/neozenith/sqlite-muninn/blob/main/examples/{name}/example.ipynb)
```

Examples that lack a `README.md` today (`llm_chat`, `llm_summarize`, `llm_tokenize`) must get one.

### R2: Dual Install Mode (PyPI Pinned vs main Branch)

Every notebook's setup cell MUST support two install modes, selectable by the user:

| Mode | Use Case | Install Mechanism |
|------|----------|-------------------|
| **PyPI pinned** | Stable demos, tutorials, workshops | `pip install sqlite-muninn==X.Y.Z` (pre-built wheel) |
| **main branch** | Development, latest features | Clone repo + `make all` + `pip install -e .` |

The setup cell presents both options with clear comments. The user uncomments the one they want. The default (uncommented) option is PyPI pinned.

### R3: Standalone Script Compatibility

The `.py` files MUST continue to work as standalone scripts with zero changes to the invocation:

```bash
python examples/semantic_search/example.py
```

This means the `.py` file is the **source of truth**, and `.ipynb` is a generated artifact.

### R4: Local Notebook Compatibility

The `.ipynb` files MUST work in local Jupyter (not just Colab). The setup cell must detect the environment and skip installation if the extension is already available.

## Architecture

### Source of Truth: Jupytext Percent Format

Each `examples/{name}/example.py` becomes a Jupytext-paired file using the `percent` format. Cell boundaries are marked with `# %%` comments in the `.py` file:

```python
# %% [markdown]
# # Semantic Search — Document Similarity with HNSW
#
# [![Open In Colab](https://colab.research.google.com/assets/colab-badge.svg)](https://colab.research.google.com/github/neozenith/sqlite-muninn/blob/main/examples/semantic_search/example.ipynb)

# %% [markdown]
# ## Setup
# Installs the muninn SQLite extension. Skip this cell if running locally with a pre-built extension.

# %%
# === Install Mode (choose one) ===

# Option A: PyPI release (default)
# !pip install sqlite-muninn==0.3.0

# Option B: Build from main branch
# !apt-get install -y libsqlite3-dev && git clone https://github.com/neozenith/sqlite-muninn.git /tmp/muninn && make -C /tmp/muninn all && pip install -e /tmp/muninn

# %%
import sqlite3
# ... rest of example ...
```

**Why percent format:**
- The `.py` file remains a valid, runnable Python script (comments are comments)
- `# %%` markers are already understood by VS Code, PyCharm, and Spyder as cell delimiters
- Jupytext generates `.ipynb` from the `.py` file deterministically
- Git diffs remain clean (`.py` is plain text)

### Extension Loading Abstraction

Currently, examples hardcode the path:

```python
# Current (dev-only)
EXTENSION_PATH = str(PROJECT_ROOT / "build" / "muninn")
db.load_extension(EXTENSION_PATH)
```

Refactor to use the existing `sqlite_muninn` package, with a fallback for standalone dev mode:

```python
# New (works in all three modes)
import sqlite3

db = sqlite3.connect(":memory:")
db.enable_load_extension(True)

try:
    import sqlite_muninn
    sqlite_muninn.load(db)
except ImportError:
    # Standalone dev mode: load from build directory
    from pathlib import Path
    PROJECT_ROOT = Path(__file__).resolve().parent.parent.parent
    db.load_extension(str(PROJECT_ROOT / "build" / "muninn"))
```

This works because:
- **Colab / pip install**: `sqlite_muninn` is installed, `loadable_path()` finds the wheel-bundled `.so`
- **Local notebook**: same as above if installed via `pip install -e .`
- **Standalone script**: `sqlite_muninn` may not be installed; fall back to the `build/` directory path

> **Note:** This is NOT graceful degradation of a requirement. Both paths load the same extension. This is environment detection — the extension MUST load or the script crashes. There is no "skip muninn" fallback.

### Environment Detection in Setup Cell

The setup cell must handle three environments without user intervention (beyond choosing install mode):

```python
import sys
import shutil

IN_COLAB = "google.colab" in sys.modules

if IN_COLAB:
    # Colab: must install — uncomment your preferred method above
    pass
else:
    # Local: check if extension is already available
    try:
        import sqlite_muninn
        print(f"muninn {sqlite_muninn.__version__} already available")
    except ImportError:
        print("sqlite_muninn not installed. Run: pip install -e . (from repo root)")
        print("Or: make all && pip install -e .")
        sys.exit(1)
```

## Colab Platform Constraints

### Linux x86_64 Only

Colab runs Ubuntu on x86_64. The muninn extension must be compiled for this target:

- **PyPI mode**: requires publishing a `manylinux` wheel with the pre-built `muninn.so`
- **main branch mode**: compiles from source on the Colab VM (has `gcc`, `make`, `libsqlite3-dev`)

### No Metal GPU

Colab has NVIDIA GPUs (CUDA), not Apple Metal. The llama.cpp build on Colab:

- Must NOT enable `GGML_METAL`
- Can optionally enable `GGML_CUDA` if the Colab runtime has a GPU
- CPU-only is the safe default for Colab

`generate_build.py` already handles this via platform detection (Darwin → Metal, Linux → no Metal). Building from source on Colab will produce the correct binary automatically.

### GGUF Model Downloads

Examples that use GGUF models (`text_embeddings`, `llm_chat`, `llm_summarize`, `llm_extract`, `llm_tokenize`) auto-download models on first run. Colab considerations:

| Model | Size | Colab Impact |
|-------|------|-------------|
| all-MiniLM-L6-v2 | 23 MB | Fast, no issue |
| nomic-embed-text-v1.5 | 146 MB | ~30s download |
| Qwen3.5-0.8B | ~600 MB | ~2 min download |
| Qwen3.5-2B | ~1.5 GB | ~5 min download |
| Qwen3.5-4B | ~2.8 GB | ~8 min download |
| Gemma-3-4B | ~2.8 GB | ~8 min download |

The `llm_extract` example downloads ALL models (~10 GB). A Colab-specific note should warn users about download times and suggest using a GPU runtime for the LLM examples.

### Memory Limits

Colab free tier: ~12 GB RAM. The `llm_extract` example loads 5 models simultaneously. This may need a Colab-specific cell that loads models sequentially or limits to a subset.

## Implementation Plan

### Phase 1: Jupytext Infrastructure

1. Add `jupytext` as a dev dependency: `uv add --dev jupytext`
2. Configure jupytext in `pyproject.toml`:
   ```toml
   [tool.jupytext]
   formats = "py:percent,ipynb"
   ```
3. Add a Makefile target:
   ```makefile
   notebooks:  ## Generate .ipynb from example .py files
   	uv run jupytext --sync examples/*/example.py
   ```

### Phase 2: Refactor example.py Files

For each of the 10 examples:

1. Add `# %%` cell markers to create notebook structure
2. Add markdown cells (`# %% [markdown]`) for title, Colab badge, section headers
3. Add the setup/install cell at the top
4. Replace hardcoded `EXTENSION_PATH` with the `sqlite_muninn` load abstraction
5. Verify the `.py` still runs standalone: `python examples/{name}/example.py`

### Phase 3: Generate and Commit .ipynb Files

**Decision: commit .ipynb or gitignore?**

| Approach | Pros | Cons |
|----------|------|------|
| **Commit .ipynb** | Colab badge works immediately; no CI needed | Binary-ish JSON diffs in PRs; merge conflicts |
| **Gitignore .ipynb + CI generate** | Clean git history; single source of truth | Colab badges only work after CI publishes to a branch; more infra |

**Recommendation: Commit .ipynb files.** Colab loads notebooks directly from the GitHub repo at the URL in the badge. If the `.ipynb` doesn't exist at that path on `main`, the badge is a dead link. Generating via CI into a separate branch (e.g., `gh-pages`) adds complexity for no user benefit.

Add a CI check that `.ipynb` files are in sync with `.py` files:

```yaml
# .github/workflows/notebooks.yml
- run: uv run jupytext --check --sync examples/*/example.py
```

This fails the build if someone edits `.py` without regenerating `.ipynb`.

### Phase 4: README Badges

For each of the 10 example directories:

1. Create `README.md` if missing (`llm_chat`, `llm_summarize`, `llm_tokenize`)
2. Add the Colab badge at the top of every `README.md`
3. Update the root `README.md` with a table of all examples + their Colab badges

### Phase 5: PyPI Wheel Publishing (Prerequisite for PyPI Install Mode)

The "PyPI pinned" install mode requires that `sqlite-muninn` is published on PyPI with pre-built wheels containing the compiled `muninn.so` for Linux x86_64.

This is a **separate workstream** (not in scope for this spec) but is listed here as a dependency. Until wheels are published:

- The default install mode in notebooks should be **main branch** (build from source)
- The PyPI option should exist but be commented out with a note: `# Requires sqlite-muninn to be published on PyPI`

### Phase 6: Makefile Integration

```makefile
notebooks:        ## Generate .ipynb from example .py files
	uv run jupytext --sync examples/*/example.py

notebooks-check:  ## Verify .ipynb files are in sync with .py sources
	uv run jupytext --check --sync examples/*/example.py
```

## Open Questions

1. **PyPI wheel timeline**: When will `sqlite-muninn` be published on PyPI with pre-built manylinux wheels? This gates the "PyPI pinned" install mode working. Until then, the "main branch" build-from-source mode is the only functional Colab path.

2. **CUDA support in Colab**: Should the main-branch build detect CUDA on Colab and enable `GGML_CUDA` for GPU-accelerated LLM inference? This would significantly speed up the `llm_*` examples but adds build complexity.

3. **Large model examples**: Should `llm_extract` (which loads ~10 GB of models) have a Colab-specific variant that uses fewer models, or should it run as-is with a warning about download times?

4. **Notebook output**: Should committed `.ipynb` files include cell outputs (showing expected results) or be output-stripped? Including outputs makes them self-documenting but increases file size and diff noise.
